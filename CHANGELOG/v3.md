## [Unreleased]

### Added

- **`p3_substantive` cadence policy is now wired live (#876).** The
  substantive-turn density policy (axis-1 option C) joins `p3_velocity`
  in the Stop and UPS dispatchers. Each turn the policy is active, the
  Stop hook classifies the user prompt via `is_substantive_turn` and
  pushes it onto a rolling window in the session ring; the firing
  predicate fires when the substantive ratio over the last
  `p3_substantive_window` turns meets `p3_substantive_threshold`. Stop
  owns the per-turn push; UPS reads the window only (a one-turn read
  lag, matching the existing `p3_velocity` counter-sharing semantics),
  so the window advances exactly once per turn. Deterministic (#605),
  no embeddings. Shadow-mode P3 capture + 4-policy scoring land in the
  next PR of the stack.

### Fixed

- **UserPromptSubmit retrieval is now conversation-aware (#909).** The
  per-prompt hook previously BM25'd the literal prompt only, so
  paraphrase / pronoun / numeric-reference follow-ups (where the topic
  vocabulary lives in the dialog history, not the current prompt) ranked
  the load-bearing thread near the bottom ŌĆö below the token-budget cut,
  never surfaced ŌĆö while noise sharing the prompt's generic words won.
  The hook now folds a small window of recent turns into the query, with
  the current prompt weighted to stay dominant. Default-on, opt-out via
  `[user_prompt_submit_hook] conversation_aware_query_enabled`; window
  and weight are tunable. Deterministic, no embeddings.

## [3.3.0] - 2026-05-21

### Added

- **P3 turn-density-aware cadence policy** ([#876](https://github.com/robotrocketscience/aelfrice/issues/876), closes the third of the [#749](https://github.com/robotrocketscience/aelfrice/issues/749) pre-registered policies). Adds `p3_turn_density` alongside the shipped P1 ([#869](https://github.com/robotrocketscience/aelfrice/pull/869)) and P2 ([#871](https://github.com/robotrocketscience/aelfrice/issues/871)) policies ŌĆö completing the [#749](https://github.com/robotrocketscience/aelfrice/issues/749) policy roster. P3's firing surface is a **transcript-byte velocity** signal: deltas in transcript file size between consecutive Stop-hook fires are bucketed into `quiet` / `normal` / `burst` classifications, and a fire is gated on a sliding window of recent classifications crossing a configurable density floor. The new `[cadence]` TOML fields are `density_window` (default 5 ticks), `density_floor` (default 0.6, fraction of ticks classified at-or-above `normal`), `quiet_threshold_bytes` / `burst_threshold_bytes` (defaults 1000 / 50000 bytes per tick); env overrides `AELFRICE_CADENCE_DENSITY_WINDOW` / `_DENSITY_FLOOR` / `_QUIET_THRESHOLD_BYTES` / `_BURST_THRESHOLD_BYTES` follow the env > kwarg > TOML > default precedence. State storage: `session_ring` gains `bytes_at_last_fire` and `recent_classifications` slots; both are forward-only and atomic-tmp-write per the existing ring pattern. Stop + UPS dispatchers now branch on `policy == "p3_turn_density"`, run the classification + density predicate over the session_ring window, and on fire write the same `cadence_resume_cache.json` that P1/P2 produce (so UPS-side resume injection per [#870](https://github.com/robotrocketscience/aelfrice/issues/870) / [#871](https://github.com/robotrocketscience/aelfrice/issues/871) works uniformly across all three policies). Default-OFF inherited from the policy-enum gating; `policy = "off"` is the ship default. Determinism per [#605](https://github.com/robotrocketscience/aelfrice/issues/605): the classification and density predicates are pure over `(byte_delta_sequence, config)`; no wall-clock or random sampling enters the firing decision. The policy slot was reserved as forward-compatible since the [#871](https://github.com/robotrocketscience/aelfrice/issues/871) cut so this lands as a value addition, not a resolver refactor.

- **Cadence shadow-evaluation mode + `aelf cadence-score` subcommand** ([#875](https://github.com/robotrocketscience/aelfrice/issues/875), blocks [#749](https://github.com/robotrocketscience/aelfrice/issues/749) campaign closure). The [#749](https://github.com/robotrocketscience/aelfrice/issues/749) P1 / P2 / P3 comparison was structurally blocked by the policy-enum mutex ŌĆö only one policy fires per project, so direct head-to-head needs a longitudinal flip-and-rebake with workload variance contaminating the comparison. Shadow mode breaks the mutex on observation only: the selected policy fires the rebuilder; the non-selected policies' fire predicates run on the same Stop-hook tick, and their fire-decisions log to a side-channel JSONL without actually firing. That yields per-turn agreement data on identical workload. New `[cadence] shadow_mode_enabled` TOML field (default `false`), env override `AELFRICE_CADENCE_SHADOW_MODE`, with env > kwarg > TOML > default resolver. Refactored `cadence.py` extracts policy-agnostic `would_fire_p1(*, fire_idx, config) ŌåÆ bool` / `would_fire_p2(*, transcript_path, last_user_prompt, config) ŌåÆ bool` predicates so Stop hook can call them against shadow configs without touching the dispatch path. New shadow-log path resolver (`<git-common-dir>/aelfrice/cadence_shadow.jsonl`), row formatter (`{ts, session_id, fire_idx, selected_policy, decisions: {p1: bool, p2: bool, p3: bool}}`), and atomic append. New `cadence_score` module computes per-policy fire-rate, the 3├Ś3 agreement matrix (pairwise per-tick decision agreement), and selected-vs-each-shadow Cohen's ╬║ over a date-range slice. New `aelf cadence-score [--since YYYY-MM-DD] [--until YYYY-MM-DD]` CLI subcommand reads `cadence_shadow.jsonl`, formats the agreement matrix + per-policy fire-rate as a human-readable table, and exits 0. The shadow path is opt-in (default-off) so a vanilla install carries zero additional disk write per Stop-hook tick. Determinism per [#605](https://github.com/robotrocketscience/aelfrice/issues/605): would-fire predicates are pure functions of `(ring state, config)`; the agreement-matrix computation is pure over the shadow log. Discretion: shadow-log carries `session_id`, `fire_idx`, and decision booleans only ŌĆö no prompt text, no transcript content. Unit tests in `tests/test_cadence_shadow.py` (predicate extraction byte-identity to dispatch path, row schema, path resolver, atomic append under concurrent writers); `tests/test_hook_stop_shadow.py` covers Stop-hook wiring; `tests/test_cadence_score.py` covers agreement matrix correctness on synthetic logs.

- **`use_intentional_clustering` ├Ś `use_type_aware_compression` compose-reconciliation** ([#878](https://github.com/robotrocketscience/aelfrice/issues/878), unblocks [#769](https://github.com/robotrocketscience/aelfrice/issues/769)). - **Stop-hook cadence policy ŌĆö P1 every-K-turns checkpoint** ([#749](https://github.com/robotrocketscience/aelfrice/issues/749)). The Stop hook now supports a periodic-checkpoint cadence that fires the rebuilder pass every K turn boundaries, independent of when the host harness chooses to PreCompact. Originating concern: post-[#746](https://github.com/robotrocketscience/aelfrice/issues/746) PreCompact-fire-rate data showed 6 fires across 2 sessions over 6 days at `trigger_mode='threshold'` ŌĆö the maintainer's typical workload `/clear`s faster than the harness compacts, so the P0 PreCompact-only policy misses most state-recovery opportunities. The [#749](https://github.com/robotrocketscience/aelfrice/issues/749) body OR-clause was tripped by maintainer self-report and this PR ships the smallest of the body's pre-registered policies. New `src/aelfrice/cadence.py` module: `CadenceConfig` dataclass, `[cadence]` TOML section (`enabled` / `policy` / `k`), env > kwarg > TOML > default resolvers, and the pure `should_fire(fire_idx, config)` predicate (`fire_idx > 0 AND fire_idx % k == 0`, gated on `enabled AND policy == "p1_every_k_turns"`). The resolver shape leaves a slot between env and kwarg for a future [#480](https://github.com/robotrocketscience/aelfrice/issues/480) `meta:hook.cadence_k` meta-belief read without changing call sites. Stop-hook wiring: a new `_maybe_fire_cadence_checkpoint(payload, session_id, serr)` helper reads the current session's `next_fire_idx` from `session_ring.read_ring_state` (same monotonic counter UPS already maintains), checks `should_fire`, and on True calls `_rebuild_and_format` with `rebuild_log_enabled=True` ŌĆö discarding the returned body since Stop has no `additionalContext` channel. The rebuild_log entry and touch-state refresh are the side effects that carry value. Stderr observability line: `aelfrice: cadence checkpoint fired @ fire_idx=N (policy=p1_every_k_turns, k=15)`. Restructured `stop()` so a session with zero lock-prompt candidates no longer early-returns before cadence gets a chance to run; the lock-prompt behaviour itself is unchanged (41 existing `test_hook_stop_lock_prompt` + `test_setup_stop_hook` tests pass). Default-OFF (`enabled = false` is the default; an unset `[cadence]` section is byte-identical to a pre-#749 Stop hook). K defaults to 15 per the [#749](https://github.com/robotrocketscience/aelfrice/issues/749) body's literature-typical starting point. Determinism contract ([#605](https://github.com/robotrocketscience/aelfrice/issues/605)): the firing predicate is a pure function of `(fire_idx, k)` ŌĆö no wall-clock, no random sampling. A replay with the same UPS sequence and same config produces the same fire decisions. 30 unit tests in `tests/test_cadence.py` cover the predicate cross-product (enabled / policy / k including k=0 / k=1 / negative-k / fire_idx=0 cold-start guard / negative fire_idx), TOML parser fail-soft on missing-file / missing-section / malformed-TOML / wrong-typed-enabled / unknown-policy / non-positive-k / bool-as-k / parent-dir walk, and the env > kwarg > TOML > default resolver precedence including env-unparseable falls-through. 11 integration tests in `tests/test_hook_stop_cadence.py` against the actual `hook.stop()` entry point cover default-OFF, policy-off-with-enabled-true, the multiples-of-K firing pattern (15/30/45 fire; 1/14/16/29 skip), cold-start fire_idx=0 suppression, missing/session-mismatched ring sentinel, empty recent-turns slice, env-override beats TOML, the stderr observability line carries policy/k/fire_idx, the regression that zero-lock-candidate sessions still fire cadence, and the lock-prompt + cadence coexistence path. P2 (token-budget watermark) and P3 (turn-density-aware) per the [#749](https://github.com/robotrocketscience/aelfrice/issues/749) body remain unimplemented; the policy enum is forward-compatible so those land as new values without rewriting the resolver or predicate.

- **Subfloor noise-pattern filter at sentence-level ingest** ([#809](https://github.com/robotrocketscience/aelfrice/issues/809), continuation of [#785](https://github.com/robotrocketscience/aelfrice/issues/785) ┬¦ 3). `_ingest_turn_ids` (the transcript / commit-ingest sentence path) now runs each candidate sentence through `_looks_like_subfloor_noise` before belief creation. Matched sentences ŌĆö code-fence boundaries (` ```bash`, `` ``` ``), header stubs ending with `:` (`"Acceptance criteria:"`, `"Pipeline composition, in order of evidence:"`), and markdown bullet stubs (`"- run tests"`, `"* foo"`, `"+ baz"`) ŌĆö do not become freestanding belief rows. When a matched sentence sits between two full-length-belief sentences within the same turn, it attaches as `anchor_text` on a new intra-turn `DERIVED_FROM` edge between the surrounding beliefs (src=later, dst=earlier, matching the inter-turn convention in `ingest_jsonl`); unanchored matches are silently dropped. Multiple sub-floor clauses between the same pair join with `" | "` and truncate to `ANCHOR_TEXT_MAX_LEN`. Closes 19% of the short-reinforced-bloat leak documented in the `retrieval-corpus-bloat` lab campaign (companion to PR #795's ┬¦1 speaker-attribution gate, which closed 51%). The pattern check is scoped to short content by a length cap (`_SUBFLOOR_MAX_LEN = 80`, the spec literal): long-form content that happens to start with `- ` or end in `:` (real prose statements, multi-sentence list items) is preserved. A pattern-only check over-applied and dropped long-form prose ending in `:`; a standalone length-floor over-applied and dropped short legit claims like `"The default port is 8080."`. The combination (pattern AND < 80) catches the load-bearing noise class while preserving both short legit claims and long-form prose. Code-fence and bullet patterns are already largely handled upstream by `extract_sentences` (paired-fence wholesale strip, line-leading list-marker strip); the gate is a backstop for malformed / mid-line cases that survive. The header-ending-in-`:` pattern is NOT handled upstream and is the load-bearing pattern in the normal pipeline. Acknowledged false-positive risk on `"He said:"` and similar ŌĆö the lab campaign named the pattern explicitly; trade-off accepted at empirical scope. Architectural deviation from spec letter (which describes the gate on "triple subject/object slots") is documented on the helper's docstring: the noun-phrase-based `triple_extractor` produces slots typically far below any length floor, so the gate has to live on the sentence-level path where the observable leak originates. 20 new tests in `tests/test_ingest_subfloor_noise.py` (four added for the length-floor boundary at 80).

### Removed

- **CONTRADICTS-edge auto-demote surface removed** ([#814](https://github.com/robotrocketscience/aelfrice/issues/814), [PR #820](https://github.com/robotrocketscience/aelfrice/pull/820)). The `Belief.demotion_pressure` schema column, the `apply_feedback(propagate=)` kwarg, the `FeedbackResult.pressured_locks` / `.demoted_locks` fields, the `aelf locked --pressured` CLI flag, and the `pressured_locks` / `demoted_locks` keys in the `aelf:feedback` / `aelf:confirm` MCP tool payloads have all been removed. The rationale is empirical and grounded in locked PHILOSOPHY [#605](https://github.com/robotrocketscience/aelfrice/issues/605) (`c06f8d575fad71fb`): "dedup/contradiction/relatedness lives in the consuming agent, not aelfrice." Across the project's lifetime, zero CONTRADICTS edges were ever recorded against locked beliefs (14 locks vs. 63,861 positive feedback rows), meaning the auto-demote walk never fired in production. A mechanism that never executes and couples the feedback path to the contradiction graph adds complexity with no measurable benefit; removal restores the central design property that `apply_feedback` is a pure Bayesian update with no graph side-effects. This is a clean removal rather than a deprecation cycle because the surface had no observed callers. **Breaking changes for callers integrating against these surfaces:** (1) `apply_feedback(propagate=False)` ŌĆö callers passing the `propagate` kwarg (notably the pre-#820 hook-search implicit-exposure path) must drop it; the kwarg no longer exists and will raise `TypeError`. (2) `FeedbackResult.pressured_locks` / `.demoted_locks` ŌĆö any code reading these fields from a `FeedbackResult` will raise `AttributeError`; the fields are not present on the returned object. (3) `aelf locked --pressured` CLI / `/aelf:locked --pressured` slash form ŌĆö the flag has been removed; passing it raises an unrecognised-argument error. (4) `aelf:feedback` and `aelf:confirm` MCP tool payloads ŌĆö the `pressured_locks` and `demoted_locks` keys are no longer included in the returned dict; consumer code indexing those keys by name will raise `KeyError` on every call.

### Performance

- **N+1 ŌåÆ set-based query for canonical-orphan count in replay drift report** ([#725](https://github.com/robotrocketscience/aelfrice/issues/725)). `_compute_replay_drift_report` previously iterated every belief id in the store, issuing one `iter_ingest_log_for_belief` full-table scan per belief to test for any non-legacy log row, then a `get_belief` fetch per orphan for the example payload ŌĆö an O(N┬▓) pattern. Replaced by a single `MemoryStore.list_canonical_orphans(limit)` call that pushes the classification into SQLite: a `NOT IN (SELECT je.value FROM ingest_log il, json_each(il.derived_belief_ids) je WHERE source_kind != 'legacy_unknown')` subquery builds the covered-belief set once; the outer SELECT walks only the orphan complement, ordered `BY b.id ASC` to preserve sample stability. Output is byte-identical (same count, same examples in the same order). Nine unit tests in `test_store_crud.py` cover the empty-store, no-log-rows, all-legacy, non-legacy-excludes, mixed-beliefs, mixed-log-rows-same-belief, limit, ordering, and content-hash-field cases.

- **Update-check cache TTL: 6h ŌåÆ 15min** (`src/aelfrice/lifecycle.py:CACHE_TTL_SECONDS`). The PyPI version-check cache used to expire after six hours, so a freshly-published release could lag the user-visible "update available" banner by up to that long on a busy machine (longer on a quiet one ŌĆö the check is gated behind the next CLI call or `UserPromptSubmit` hook fire). PyPI's JSON endpoint is CDN-cached and unauthenticated, so a 15-minute cadence is well within the polling-etiquette band and shrinks the worst-case banner-lag window from 6h to ~15min. Detached-subprocess + on-disk-cache architecture is unchanged.

### Fixed

- **`merge-train` now pre-checks conversation resolution before FF push** ([#721](https://github.com/robotrocketscience/aelfrice/issues/721)). PRs with unresolved review threads (Sourcery, CodeRabbit, or human) would pass all five merge-train gates and then fail at the push step with `GH006: Protected branch update failed ŌĆö All comments must be resolved`, wasting a full 10-minute CI poll and leaving the operator to manually identify which threads to resolve. A new `[3/6] conversation-resolution check` step queries the PR's `reviewThreads` via GraphQL (`isResolved==false AND isOutdated==false`), collects the unique file paths of unresolved threads, and `fail_and_unlabel`s with a message listing those paths before the CI poll starts. The existing 5-step sequence (1/5..5/5 + out-of-band 6/6) is renumbered to 1/6..6/6 + 7/7. The check uses the same `fail_and_unlabel` helper and `<!-- merge-train-v1 -->` marker so operators see a clear, actionable message instead of a raw push error.

- **`Store.update_belief` SET clause is now full-row** ([#708](https://github.com/robotrocketscience/aelfrice/issues/708)). The docstring promised "Full-row update" but the `UPDATE beliefs SET ...` clause omitted four columns that `insert_belief` writes: `hibernation_score`, `activation_condition`, `retention_class`, `valid_to`. Callers that mutated any of those fields and round-tripped through `update_belief()` would observe the change in-memory but not on disk ŌĆö the next `get_belief()` returned the stale row. Latent today (the existing call sites either don't touch those columns or go through dedicated paths like `set_retention_class()` / `soft_delete_belief()`), but `#689`'s `aelf promote --to-scope` and any future caller relying on the docstring would have hit it. SET clause now covers all 17 dataclass-mapped columns; `retention_class` validation mirrors `insert_belief` (rejects values outside `RETENTION_CLASSES` with `ValueError` before any UPDATE). Two new round-trip tests in `tests/test_store_crud.py` pin the property: one mutates each previously-missing column and asserts on-disk persistence, one asserts the new validation fires and leaves the existing row untouched. Surfaced by the post-rebase reviewer on PR #694.

- **`merge-train` now auto-closes linked issues after a successful FF push** ([#634](https://github.com/robotrocketscience/aelfrice/issues/634)). GitHub's auto-close-on-merge behavior fires only when a merge *commit* lands on the default branch; the merge-train workflow does an FF push (required by this repo's signed-commit branch-protection model ŌĆö the bot has no signing key, so `gh pr merge` can't be used), so GitHub never saw a merge event and never parsed `Closes #N`. Reviewers had been manually closing four issues per day. The post-FF step now reads the merged PR body (capped to the first 8KB), greps case-insensitively for `Closes #N` / `Fixes #N` / `Resolves #N`, dedupes the matches, and runs `gh issue close <N> --reason completed --comment "Shipped via PR #M (merged <SHA>)."` for each. Idempotent ŌĆö already-closed issues are skipped after a state probe; failures log to stderr but do not fail the workflow because the merge has already landed. Cross-repo references (`closes other/repo#5`) are intentionally out of scope.

- **`merge-train` treated CANCELLED check runs as failures and did not deduplicate by name** ([#632](https://github.com/robotrocketscience/aelfrice/issues/632)). When CI re-ran a check on the same SHA (older invocations getting cancelled by the newer trigger), the bot evaluated every historical run on the SHA ŌĆö so the cancelled rows tripped the failure branch even when the latest run for that check name was `success`. PR #614 hit this twice (`required check(s) failed: size-check, size-check` while a later `size-check` was green); PR #631 was about to hit the same. Rollup is now grouped by `name` and reduced to the latest run via `max_by(.started_at)` before evaluation, mirroring how GitHub's branch-protection required-checks rule works downstream. `cancelled` is also dropped from the failure conclusions list ŌĆö under the new dedup the only way a `cancelled` row reaches evaluation is if no successor ran, and that's better surfaced by the existing `pending` / timeout paths than reported as a hard failure.

- **`merge-train` signature check rejected every signed PR** ([#619](https://github.com/robotrocketscience/aelfrice/issues/619)). The check shipped in #602 used `git log --format='%G?'`, which only registers SSH signatures as `G` when the runner has `gpg.format=ssh` and `gpg.ssh.allowedSignersFile` configured ŌĆö `actions/checkout` sets neither, so every SSH-signed commit fell through to GPG parsing and `%G?` returned `N`. Every PR labeled `ready-to-merge` since #602 shipped was unlabeled with a spurious "not signed" comment, and `main` only advanced via manual operator FF pushes. Replaced with `gh api repos/.../commits/<sha> --jq '.commit.verification.verified'`, the same source of truth the `required_signatures` branch protection rule uses downstream. Any commit the workflow accepts will also pass the push gate.

- **`/aelf:upgrade` slash no longer stops after detection** ([#611](https://github.com/robotrocketscience/aelfrice/issues/611)). Rewrites `src/aelfrice/slash_commands/upgrade.md` so the agent reading the slash cannot reasonably stop after step 1 (detection) and treat the CLI's parenthetical advisory (`(installed via pipx ŌĆö use pipx to upgrade)`) as a substitute for actually running the upgrade. Concrete edits: objective opens by naming the failure mode (\"Producing the detection output and stopping is the wrong outcome ŌĆö that is what `aelf upgrade-cmd` does on its own. This slash exists to take the next step.\"); step 2 instructs the agent to run the captured `<command>` *verbatim* and labels it \"the operation the slash was invoked to perform\"; a new `<failure_modes>` section calls out the two anti-patterns observed in #611 (stop-after-step-1, and substituting a \"simpler\" command guess). Adds regression test `test_upgrade_slash_keeps_step_2_imperative` locking in the imperative load-bearing wording. CLI behavior unchanged ŌĆö detection stays advisory by design (in-process package replacement is unsafe on Windows); the slash is what makes the upgrade happen.

- **`merge-train` re-fires on PR push when `ready-to-merge` is already set** ([#646](https://github.com/robotrocketscience/aelfrice/issues/646)). The serialized merger added in #602 only listened for the `labeled` event, so a PR that was force-pushed (e.g. after a local rebase) while still carrying `ready-to-merge` never re-triggered the workflow ŌĆö the operator had to flip the label off and back on to wake it up. The `pull_request` trigger now includes `synchronize` and `edited` plus a guard that checks for the `ready-to-merge` label on each fire, so a rebased branch with the label still attached merges as soon as CI clears the new SHA.

- **Compound-confidence fork-tie threshold is ratio-based** ([#668](https://github.com/robotrocketscience/aelfrice/issues/668)). Follow-up to the #658 R2 fork-on-CONTRADICTS work: the prior tie detector compared absolute confidence values and falsely tagged two paths as a tie when their scores happened to be near-equal floats with very different magnitudes. The detector now compares the ratio of the second-best to best path confidence against a fixed threshold, which is scale-invariant and matches the operator-intuition that "within 5% of each other" is a tie regardless of whether the best score is 0.85 or 0.12.
