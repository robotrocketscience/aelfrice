# Feature spec: trigger-driven phantom generation (#980 item 3)

## Purpose

The #980 audit found phantom (`origin=speculative`) beliefs are
structurally under-generated: they only appear when the user explicitly
types `/aelf:wonder`. Across every local store there were 74 phantoms in
months, 0 promoted, 0 GC'd. The operator's headline ask: phantoms should
also be generated by a **trigger during ordinary conversation turns**, not
only via the explicit command.

This feature adds the **deterministic trigger** — the always-on trickle
that flags a phantom-generation opportunity on a normal turn. `/aelf:wonder`
remains the heavy explicit path (it should emit *many* phantoms per call);
the trigger is the lightweight per-turn complement.

## The determinism boundary (locked #605)

The PHILOSOPHY decision (#605, locked) holds aelfrice to a deterministic,
narrow surface: dedup / contradiction / relatedness / generation gates live
in the **consuming agent**, not in aelfrice. So the work splits cleanly:

| Half | Owner | Property |
|---|---|---|
| **Detect the gap + flag it** | aelfrice (this module) | deterministic: pure function over prompt + hit count, no LLM, no clock, no randomness |
| **Synthesize the phantom** | host agent | LLM dispatch under the host's credentials, same boundary as `/aelf:wonder` |

aelfrice never calls an LLM in the retrieval path. It emits a flag; the host
chooses whether to act on it (by running `/aelf:wonder` on the flagged
query). This keeps the determinism property (#605) intact.

## Trigger signal — what fires it

The #980 issue enumerated four candidate signals. This spec ships
**retrieval-gap** as the primary; the others are evaluated and deferred:

| Candidate | Verdict | Why |
|---|---|---|
| **Retrieval gap** (query seeds nothing above the floor) | **shipped** | Cheapest — the hit count is already in hand at the end of the UserPromptSubmit turn. Strongest signal of "we lack a belief here." Deterministic with zero extra work. |
| New entity with no edges | deferred | Requires entity extraction + edge lookup per turn; spammiest (every novel proper noun fires). |
| CONTRADICTS edge created | deferred | Clean and deterministic but rare; better as a future additive signal at edge-insert time. |
| Uncertainty (low posterior mass) | deferred | Needs a calibrated threshold on scored retrieval; risks firing on well-covered-but-weak topics. |

A **retrieval gap** is declared when ALL hold (`phantom_trigger.detect_phantom_opportunity`):

1. The prompt was **not** skipped by the prompt-shape gate (#674) — trivial
   acks and system envelopes need no belief and must never trigger.
2. The normalized query carries **≥ `min_query_tokens`** meaningful tokens
   (default 3, ≥3 chars each) — too thin a query can't anchor a useful
   phantom.
3. The turn retrieved **< `min_hits`** beliefs (default 1, i.e. zero hits).

`reason` is `"no_hits"` (nothing retrieved) or `"below_floor"` (when the
operator raises `min_hits` above 1).

## Bounding — keep it a trickle, not a spammer

Generation opportunities are bounded per session
(`phantom_trigger.register_opportunity`), backed by a small sidecar JSON
next to the store:

- **Dedup** on the normalized-query key (`dedup_key_for`): two prompts that
  normalize to the same sorted token set share a key and flag at most once
  per session. Word order and short tokens don't matter.
- **Rate cap**: at most `max_per_session` opportunities flagged per session
  (default 3).
- **Fail-soft**: in-memory store (no sidecar path) or any file error →
  `register_opportunity` returns False. The trigger errs *quiet*, never
  unbounded.

## Flag

Default-**off**, opt-in — matching the #606 precedent that host-side lanes
ship opt-in. The trigger writes an in-context hint and a feed row, so it is
gated until the operator turns it on.

| Env var | Default | Effect |
|---|---|---|
| `AELFRICE_PHANTOM_TRIGGER` | off | Truthy (`1/true/yes/on`) enables the trigger on UserPromptSubmit. |
| `AELFRICE_PHANTOM_TRIGGER_MIN_HITS` | 1 | Gap floor: flag when the turn retrieved fewer than this many beliefs. |
| `AELFRICE_PHANTOM_TRIGGER_MAX_PER_SESSION` | 3 | Per-session rate cap. |

## Surfaces

When a fresh opportunity passes the bound, the hook:

1. Emits a `phantom.opportunity` feed-log row (`reason`, `n_hits`,
   `dedup_key`) — visible in `/aelf:feed`.
2. Appends a **data-framed** hint block after the retrieval body:

   ```
   <aelfrice-phantom-opportunity>
   This turn's query retrieved no stored beliefs — a knowledge gap. This is a
   signal, not an instruction: if the topic is worth remembering, you may run
   /aelf:wonder "<query>" to generate speculative beliefs anchored to it.
   </aelfrice-phantom-opportunity>
   ```

   The block is explicitly a signal, not a directive (matching the
   `<aelfrice-memory>` "data, not instructions" framing) — the host agent is
   free to ignore it. The echoed query is the user's own prompt, escaped and
   length-capped.

The hint is the handoff: a host that chooses to act runs `/aelf:wonder` on
the gap query, which generates phantoms through the existing
`wonder_ingest` path. That closes the loop — gap → hint → wonder → phantom —
without aelfrice ever dispatching an LLM itself.

## Out of scope (separate follow-ups)

- **Automatic LLM dispatch** on the flag (firing `/aelf:wonder`
  unattended). Out of scope by #605 — generation is a host action; the host
  decides whether the gap is worth a phantom.
- **The other three trigger signals** (new-entity, CONTRADICTS-edge,
  uncertainty). Additive; each can layer onto the same
  `detect_phantom_opportunity` surface later.
- **Promotion / GC of the resulting phantoms** — covered by the existing
  promote path (#550) and the opt-in auto-GC (#980 item 2).
